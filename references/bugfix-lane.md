# Bugfix Lane

Use this lane for bounded defects where the expected behavior is known and the
fix can be verified without creating a full feature spec.

## Flow

```text
Investigate -> RED -> Fix -> Audit -> Quality Gate -> Done
```

## 1. Investigate

- Reproduce or localize the defect before changing production code.
- Identify the smallest behavior claim that would prove the bug is fixed.
- Choose `strict_tdd` when the bug is in a tested code path or a focused test
  can be written at reasonable cost.
- Escalate to Feature-Spec Lane or Full Harness when the defect spans unclear
  requirements, multiple owners, migration work, broad UI flow changes, or
  unresolved architecture decisions.

## 2. RED

- Add or run a focused failing test, fixture, trace check, or executable
  reproduction that fails for the bug being fixed.
- Record the command, result, failure reason, and evidence location before the
  fix.
- Do not claim a bug is fixed without failure evidence. A test that only runs
  after the fix is regression verification, not TDD evidence.
- If no executable RED path is available, stop and either request a decision or
  downgrade the claim to investigation/workaround with a concrete substitute
  reason. Do not label the outcome as a verified bug fix.

## 3. Fix

- Make the smallest code or configuration change that addresses the RED
  evidence.
- Keep unrelated refactors out of the bugfix lane unless they are necessary for
  the fix and covered by the same verification path.
- Append trace events or report evidence as the work proceeds; do not rely on a
  final narrative to reconstruct chronology.
- Prefer running tests through `scripts/harness_test_run.py` so RED/GREEN evidence
  is generated by the wrapper rather than hand-written by the worker.

## 4. Audit

- Review the diff against the investigated defect and confirm the change maps
  to the failing evidence.
- Check boundary conditions, error paths, compatibility, and whether the fix
  broadens behavior beyond the intended defect.
- If the audit reveals unclear product behavior or a wider design problem,
  escalate before continuing.

## 5. Quality Gate

- Run the original RED command again and record the GREEN result.
- Run relevant surrounding tests, lint/typecheck/build, or the smallest broader
  regression check justified by the touched surface.
- For Full Harness or delegated work, run the TDD trace checker when available:
  `python3 scripts/tdd_gate_check.py <path-to-tdd_trace.jsonl>`.
- When validating physical chronology, include changed source files:
  `python3 scripts/tdd_gate_check.py --source-path <file> <path-to-tdd_trace.jsonl>`.
- A substitute gate is valid only for non-behavior, docs/config-only, or
  genuinely disproportionate test paths, and it must include a no-test reason.

## Retry, Checkpoint, And Rollback

- Limit a single bugfix RED/GREEN loop to a small retry budget, normally three
  attempts.
- Store the latest failure scene in `run_state.json.tdd_current_cycle_context`
  when using the wrapper.
- After a verified GREEN in an isolated worktree, create a local checkpoint
  commit if the next refactor or test cycle is risky.
- Do not run `git reset --hard` automatically in the main worktree. Use hard
  rollback only inside an isolated disposable worktree or after explicit user
  authorization.

## 6. Done

Done requires all of the following:

- The report includes `Test-First Or Substitute Verification`.
- RED evidence predates the fix evidence in the trace or command chronology.
- GREEN and any surrounding checks pass or are explicitly reported as blocked.
- The audit found no unverified critical path, stub, TODO, mock, or unrelated
  behavior change.
